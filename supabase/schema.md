# Database Schema

Human-readable companion to [schema.sql](schema.sql). If you change the schema, update both.

**Project:** `HardCoders` (`vbimsbasupcbdxzsazif`)
**Postgres:** 17.6
**Migrations applied:** `initial_schema`, `harden_function_security`, `revoke_definer_grants_from_client_roles`

## Tables

### `profiles`

One row per user. Mirrors Supabase's `auth.users` table with our own app-level fields.

| Column | Type | Nullable | Default | Notes |
|---|---|---|---|---|
| `id` | `uuid` | no | — | **PK**, FK to `auth.users(id)` with `on delete cascade`. Same id as the auth user. |
| `display_name` | `text` | yes | — | Auto-populated from `raw_user_meta_data.display_name` or `email` on signup. |
| `role` | `text` | yes | — | Job role (e.g. "Frontend", "Backend"). User-editable. **Distinct from `memberships.role`.** |
| `avatar_url` | `text` | yes | — | Optional. |
| `created_at` | `timestamptz` | no | `now()` | |

**Why it exists:** Supabase Auth owns the password / session machinery in `auth.users`. We mirror the id in `public.profiles` so RLS policies on app data can reference `auth.uid()` cleanly without crossing schemas, and so we can store app-level fields without polluting the auth table.

**How rows are created:** Automatically by the `handle_new_user` trigger when `auth.users` gets a new row (i.e. when someone signs up). Never insert directly.

---

### `teams`

A team is the "circle" — the unit that members share standups, blockers, and availability with.

| Column | Type | Nullable | Default | Notes |
|---|---|---|---|---|
| `id` | `uuid` | no | `gen_random_uuid()` | **PK** |
| `name` | `text` | no | — | Display name. Not unique — two teams can share a name. |
| `join_code` | `text` | no | *(trigger)* | 6-digit numeric, **unique**. Auto-generated by `set_team_join_code` trigger on insert. Permanent. |
| `created_at` | `timestamptz` | no | `now()` | |

**Why it exists:** Standups and blockers belong to a team, not a user. Multiple teams can coexist (a user might be in a class team and a startup team, etc.).

**Why `join_code` is on `teams` and not on a separate `invites` table:** Life360-style — one permanent shareable code per team. Simpler than invite tokens. If a code leaks, "regenerate code" is a future feature (deferred).

**Tiering note:** No `tier` column yet. Adding one later is non-breaking — give it a default like `'group_project'` and existing rows pick that up.

---

### `memberships`

Many-to-many join table between `profiles` and `teams`.

| Column | Type | Nullable | Default | Notes |
|---|---|---|---|---|
| `user_id` | `uuid` | no | — | FK to `profiles(id)`, `on delete cascade`. |
| `team_id` | `uuid` | no | — | FK to `teams(id)`, `on delete cascade`. |
| `role` | `text` | no | `'member'` | Check constraint: `'member' | 'lead' | 'manager'`. **Team role, not job role.** |
| `joined_at` | `timestamptz` | no | `now()` | |

**Primary key:** `(user_id, team_id)` — a user can be in a team only once.

**Why it exists:** Supports many-to-many — one user in multiple teams, one team with multiple users. Per-team roles (lead vs member) are stored here rather than on `profiles` because the role is contextual to the team.

## Functions

### `generate_join_code() → text`
Picks a random 6-digit numeric code, retries on collision against existing `teams.join_code` values. Internal — used by the trigger. Search path pinned to `public`.

### `set_team_join_code()` *(trigger function)*
Fires `before insert on teams`. If `join_code` was not provided, fills it from `generate_join_code()`. So client code just inserts `{ name }`; the code appears for free.

### `join_team_by_code(p_code text) → teams`
The **canonical way for a signed-in user to join a team**. Atomic: looks up the team by code, inserts a `memberships` row for `auth.uid()`. Raises `invalid_join_code` (SQLSTATE `P0001`) if the code doesn't match.

`security definer` so it can read `teams` even though the calling user isn't a member yet (RLS would otherwise hide the row). EXECUTE granted to `authenticated` only — anon callers are rejected.

Use from the client:
```js
const { data, error } = await supabase.rpc('join_team_by_code', { p_code: '123456' });
// data → team row, or error.code === 'P0001'
```

### `handle_new_user()` *(auth trigger function)*
Fires `after insert on auth.users` (i.e. on signup). Creates a matching `profiles` row. `security definer` because the calling context is the auth system, not the new user.

EXECUTE revoked from `public / anon / authenticated` — only the trigger should invoke it. (Triggers don't depend on EXECUTE grants.)

## Row-Level Security

RLS is enabled on every table. Default-deny: queries return empty unless an explicit policy permits them.

### `profiles`
- **`profile_self_read`** — `select` your own profile (`auth.uid() = id`).
- **`profile_teammate_read`** — `select` profiles of users you share a team with. Powers "see your teammates" on the dashboard.
- **`profile_self_update`** — `update` your own profile only.

No `insert` policy: profiles are created exclusively by the `handle_new_user` trigger.

### `teams`
- **`team_member_read`** — `select` teams you belong to.
- **`team_authed_insert`** — any signed-in user can `insert` a new team.

**Known gap:** after `INSERT`, the user can't read the row back via `.select()` because they're not yet a member. The fix is a `create_team(p_name)` RPC that does insert-team + insert-membership atomically — not yet implemented.

### `memberships`
- **`membership_self_read`** — read your own memberships and those of your teammates (so you can see who's on a shared team).
- **`membership_self_insert`** — insert only memberships where `user_id = auth.uid()` (you can join yourself, not others).

## Common operations (client-side)

### Sign up
```js
const { data, error } = await supabase.auth.signUp({
  email, password,
  options: { data: { display_name: 'Shazi' } },
});
// profile row auto-created by trigger.
```

### Sign in
```js
const { data, error } = await supabase.auth.signInWithPassword({ email, password });
```

### Join a team by code
```js
const { data: team, error } = await supabase.rpc('join_team_by_code', { p_code: '123456' });
// error.code === 'P0001' → "invalid code"
```

### Create a team *(currently has the gap noted above)*
Eventually:
```js
const { data: team, error } = await supabase.rpc('create_team', { p_name: 'Sprint 5 Team' });
```

For now (workaround):
```js
const { data: team } = await supabase
  .from('teams')
  .insert({ name: 'My Team' });
// Then a separate select by id, or insert into memberships first.
```

## How to extend the schema safely

1. Edit [schema.sql](schema.sql) with the new tables / columns / policies.
2. Apply a named migration via the Supabase MCP (or the CLI) — don't paste straight into the Dashboard. Migrations are tracked; arbitrary SQL is not.
3. Run the Supabase security advisor; resolve or document any new warnings.
4. Update this doc with the new entries.
5. PR the schema + doc together so reviewers can see both at once.
